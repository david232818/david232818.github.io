---[ 1. Out Of Bound 취약점
 [1]은 많은 C 구현체에서 auto 타입으로 선언된 배열의 끝을 넘어선 크기의
데이터를 씀으로써 실행 스택을 부수는 것이 가능함을 보였다. 이를 가능하도록
만드는 것은 gets, strcpy 함수 등과 같은 메모리 쓰기 프리미티브이다. 즉,
이러한 메모리 쓰기 프리미티브를 통해 배열의 크기를 넘어서는 스택의 다른
부분까지 덮어쓸 수 있고, 덮어쓰는 부분이 (리턴 주소와 같이) 실행 흐름에
관여한다면 이를 조작할 수 있다는 것이다. 이러한 문제가 발생할 수 있는
이유는 gets, strcpy 함수 등이 [2, 3]이 설명하듯이 메모리의 경계 검사를
수행하지 않고 특정 데이터가 나올 때까지 작업을 수행하기 때문이다.

 여기서 다루는 Out Of Bound 취약점도 같은 맥락에서 볼 수 있다. 이 취약점도
메모리의 경계 검사를 수행하지 않기 때문에 발생하지만, 메모리에 접근하도록
만드는 프리미티브는 메모리의 인덱스를 사용자로부터 입력을 받거나 조작이
가능하고 이때 경계 검사를 수행하지 않는 코드가 된다. 이러한 코드는 함수가
될 수도 있고, 한 줄이 될 수도 있다. 예를 들어 다음과 같은 C 코드의 조각을
보자.

-------------------------------------------------------
...
    while (true)
    {
        size = get_size("size: ");
        index = get_idx("index: ");

        buf = malloc(size);
        gets(&buf[index]);
    }
...
-------------------------------------------------------

위 코드에서는 index 변수를 통해 입력을 받아들일 buf에서의 위치를 정한다.
그런데 index 값에 대한 검사를 수행하는 코드는 존재하지 않는다. 따라서
사용자가 index 값으로 어떤 값을 입력하든 프로그램은 그 위치부터 입력을
읽어들인다.

---[ 2. R/W primitives
 1.에서 OOB (Out Of Bound) 취약점이 메모리 접근 프리미티브가 될 수 있음을
확인하였다. 여기서 puts, printf 함수 등과 같은 메모리 읽기 프리미티브를 통해
메모리의 값을 읽을 수 있고, gets, strcpy 함수 등과 같은 메모리 쓰기
프리미티브를 통해 메모리에 값을 쓸 수 있다.

 메모리 읽기 프리미티브를 사용하는 경우는 다른 영역의 메모리 값을 읽을 때이다.
다른 영역의 메모리에는 민감한 데이터가 존재할 수도 있기 때문이다. 하지만
더 구체적인 경우는 동적으로 링크된 라이브러리 함수의 주소를 얻고자 하는
때이다. 이는 함수의 주소가 실행 시마다 달라지는 경우에 라이브러리의 시작
주소를 얻어 (system 함수와 같이) 호출하고자 하는 라이브러리 함수의 주소를
계산하기 위해서이다.

 1.의 예시 코드에서 사용자는 buf 뒤에 존재하는 유효한 메모리 영역 중 쓰기
권한이 있는 영역에 데이터를 쓸 수 있게 된다. 이때 메모리 쓰기 프리미티브는
gets 함수가 된다. 여기서 만약 사용자가 입력한 index 값이 리턴 주소를
가리킨다면 리턴 주소의 값을 변경할 수 있을 것이고, 함수 포인터를 가리킨다면
후에 호출될 함수를 변경할 수 있을 것이다.
